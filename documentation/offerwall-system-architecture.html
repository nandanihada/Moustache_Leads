5. User Flows

Flow A: User Opens Offerwall → Enters Sub-Wall

1. User lands on offerwall.moustacheleads.com
2. Offerwall loads → checks offerwall_settings → applies pinned offers, hidden offers, order
3. User sees: [Pinned Offers] → [Sub-Wall Cards] → [Router Cards] → [Regular Offers]
4. User clicks a Sub-Wall card (e.g., "Premium Surveys")
5. Opens walls.moustacheleads.com/premium-surveys
6. If pre_screening_enabled → show PreScreeningForm (age, gender)
7. User submits → backend checks filter_rules → returns eligible offers
8. User sees filtered offer cards inside the sub-wall

Flow B: User Clicks Router Card → Survey Routing

1. User clicks Router card on offerwall
2. POST /api/router/:id/start → creates router_session
3. System picks Lane 1, Survey Position 1
4. Shows pre-qual questions for Survey #1 (e.g., "Are you 18-35?")
5a. If PASS → redirect user to actual survey URL (PepperAds etc.)
5b. If FAIL → auto-advance to Survey #2 pre-qual questions
6. User is on external survey...
7a. Survey COMPLETE → callback hits /api/router/:id/callback → show Congrats page
7b. Survey DQ (disqualified mid-way) → callback → auto-route to next survey
7c. TIMEOUT (too long) → frontend timer fires → POST next → route to next
8. After congrats, user clicks "Next" → repeat from step 3 for next survey
9. If all surveys exhausted → show "No more surveys available"

Flow C: Admin Creates a Sub-Wall

1. Admin goes to /admin/offerwall-manager/sub-walls
2. Clicks "Create Sub-Wall"
3. Fills: name, slug, image, selects offers from dropdown
4. Toggles "Enable Pre-Screening" → picks/creates a survey
5. Sets filter rules: "If age 18-24, show offers A,B. If 25-34, show A,B,C"
6. Saves → sub-wall appears as card on main offerwall
7. Also accessible directly at walls.moustacheleads.com/{slug}

Flow D: Admin Creates a Router

1. Admin goes to /admin/offerwall-manager/routers
2. Clicks "Create Router"
3. Names it, adds Lane 1
4. Drags surveys into Lane 1 in order (Survey A → B → C)
5. For each survey: sets pre-qual criteria (age range, gender) + timeout
6. Toggles "Show on Offerwall" or assigns to a specific sub-wall
7. Saves → router card appears on wall

6. Build Order & Dependencies

Phase 1 Offerwall Manager

Depends on: Nothing (standalone)

Build:

  • offerwall_settings collection + model
  • 6 admin API endpoints
  • AdminOfferwallManager page with iframe + controls
  • Modify existing offerwall route to read settings

Est: 2-3 days

Phase 2 Survey System

Depends on: Phase 1 (lives as subtab)

Build:

  • surveys + survey_responses collections
  • 8 API endpoints (admin + public)
  • SurveyBuilder admin page
  • SurveyRenderer component (reusable)
  • AI generation endpoint (optional, can add later)

Est: 3-4 days

Phase 3 Sub-Walls

Depends on: Phase 2 (reuses survey engine for pre-screening)

Build:

  • sub_walls + pre_screening_responses collections
  • 7 API endpoints
  • AdminSubWalls page
  • SubWallCard component (for main offerwall)
  • SubWallPage (public page with pre-screening)
  • New subdomain/route: walls.moustacheleads.com

Est: 4-5 days

Phase 4 Redirect Router

Depends on: Phase 2 + Phase 3

Build:

  • routers + router_sessions collections
  • 10 API endpoints
  • AdminRouters page with lane editor
  • RouterCard component
  • RouterFlow page (qualification + routing UI)
  • Callback handler for DQ/complete from providers
  • Timeout detection (frontend timer)

Est: 5-7 days

Total Estimated: 14-19 days

Build Sequence:

Phase 1 (Offerwall Manager) → can start immediately
    ↓

    ↓
Phase 3 (Sub-Walls) → needs survey engine from Phase 2
    ↓
